## 1. 하이브리드 모델 라우팅 구현

### 배경

gpt-4o-mini는 비용·속도에서 유리하지만, "~한 내용이 있나요?" 같은 존재 여부 질문에서 전제를 검증하지 않고 답변을 생성하는 경향이 있다.

### 설계

```
일반 질문           → gpt-4o-mini
존재 여부 질문      → gpt-4o  (needs_reasoning_model=True)
소관 외 주제 질문   → gpt-4o
```

**감지 로직** (`service/llm/prompt_templates.py`):

- `_is_existence_query()`: regex 패턴으로 "논의된 적이 있나요?", "한 내용이 있나요?" 등 감지
- `_is_out_of_scope()`: 국민연금, 조세정책, 검찰개혁 등 소관 외 키워드 감지
- `needs_reasoning_model()`: 위 두 함수의 OR

### 2단계 존재 검증

존재 여부 질문은 생성 전 verifier를 먼저 실행:

```python
# generate.py
if _is_existence_query(question):
    if not _verify_claim(question, context):
        state["draft_answer"] = "회의록에서 해당 내용은 확인되지 않았습니다."
        state["generation_skipped"] = "existence_not_confirmed"
        return state
```

`_verify_claim()`: gpt-4o에 CONFIRMED / NOT_CONFIRMED 이진 판단 요청 (max_tokens=20)

## 2. REFUSED grounding 레벨 신설

기존 NONE(근거 없음)과 올바른 거절을 구분할 수 없어 UX 혼란이 있었다.

| 레벨 | 조건 | UI 배지 |
|---|---|---|
| FULL | score > 0.6 | 녹색 "근거 충분" |
| PARTIAL | 0 < score ≤ 0.6 | 노란색 "일부 근거" |
| NONE | score = 0, 일반 | 빨간색 "근거 부족" |
| **REFUSED** | score = 0, 거절 문구 감지 | 회색 "확인 불가" |

REFUSED 감지 조건 (`grounding_check.py`):
- `level == "NONE"` AND
- 답변에 `[n]` 없음 AND
- `## 메인 결과` 섹션에 "확인되지 않", "확인할 수 없" 등 패턴 포함

### 원인 추적

`_EXISTENCE_PATTERNS`의 광범위한 패턴이 정상 질문을 오분류:

```
"어떤 발언을 했나요?" → r"했나요\?" 매칭
"차이가 있나요?"     → r"있나요" 매칭
"지적했나요?"        → r"했나요\?" 매칭
```

오분류 발생 시 연쇄 실패:
1. existence check 발동 → `_verify_claim()` 호출
2. NOT_CONFIRMED → 생성 스킵 → 헤더 없는 plain 거절 답변 생성
3. `_extract_conclusion_text()`가 `## 메인 결과` 헤더를 못 찾아 빈 문자열 반환
4. REFUSED 감지 실패 → grounding = NONE
5. fallback citations 5개 붙음 (거절인데 관련 없는 인용 노출)

영향 문항: eval_001, eval_003, eval_004, eval_008, eval_011, eval_016, eval_039, eval_046 등 12건 이상

### 수정

`_EXISTENCE_PATTERNS` 구체화 (`service/llm/prompt_templates.py`):

```python
# 제거
r"있나요"    # 너무 광범위
r"했나요\?"  # 모든 의문문 매칭

# 유지/추가 (의미 단위 콤보)
r"한\s*내용이\s*있나요"
r"발언한\s*내용이\s*있"
r"논의된\s*적이\s*있나요"
r"한\s*적이\s*있나요"
r"말했나요"
r"있었나요"
r"있었습니까"
```

eval 스크립트 수정 (`eval/run_eval.py`):
```python

# 변경 후
refused = ... and grounding in ("NONE", "REFUSED")
```

## 4. 교훈

- **eval 없이는 회귀를 알 수 없다.** 하이브리드 모델이라는 기능을 추가했는데 오히려 grounding이 절반으로 떨어졌다. 숫자로 측정하지 않았으면 "잘 된다"고 생각했을 것.
- **패턴이 넓을수록 조용히 실패한다.** `r"했나요\?"`는 단 하나의 패턴이지만 수십 개 질문에 영향을 미쳤다. 디버깅 단서는 grounding_check의 헤더 의존성이었다.
- **새 레벨(REFUSED)을 추가할 때 기존 채점 스크립트도 같이 수정해야 한다.** eval 스크립트가 `grounding == "NONE"`만 체크해서 REFUSED 거절을 실패로 집계했다.
