## 0. 프로젝트 한 줄 정의

국회 회의록을 기반으로 정책 의제, 행위자, 쟁점, 입장 차이, 시계열 흐름을 분석하고, 모든 답변을 원문 근거와 연결하는 GovTech RAG 서비스.

## 1. 전체 개발 흐름

이 프로젝트는 다음 순서로 진행한다.

```text
서비스 기획
→ 서비스 설계
→ 개발
→ 소스관리
→ 배포
→ 출시
→ 유지보수
```

각 단계의 역할은 다음과 같다.

```text
기획:
무엇을 만들고 누구에게 제공할지 정한다.

설계:
화면, 데이터, API, RAG 흐름, 신뢰 장치를 도면화한다.

개발:

내 컴퓨터가 아니라 실제 사용자가 접속할 수 있게 서비스화한다.

출시:
서비스 링크, README, 데모 질문, 포트폴리오 설명을 갖춰 공개한다.

유지보수:
출시 후 검색 품질, 답변 신뢰도, 데이터, 비용, 오류를 계속 관리한다.
```

## 2. 서비스 기획

### 2-1. 기획의 의미

기획은 "무엇을 만들 것인가"를 정하는 단계다.

이 프로젝트에서는 단순한 회의록 검색 챗봇이 아니라, 국회 회의록을 정치학·정책학 관점에서 분석하는 GovTech 서비스를 목표로 한다.

### 2-2. 서비스 목적

국회 회의록은 정책 결정 과정, 정당 간 쟁점, 정부 답변, 의원별 문제의식이 축적된 중요한 공적 자료다. 하지만 분량이 많고 검색 중심으로 제공되어 사용자가 빠르게 맥락을 파악하기 어렵다.

이 서비스의 목적은 국회 회의록을 단순 검색 대상이 아니라 정치학·정책학 분석 자료로 활용할 수 있게 만드는 것이다.

### 2-3. 타깃 사용자

- 정책 실무자: 국회의원, 보좌관, 공무원
- 미디어 실무자: 기자, 방송·언론 관계자
- 학생: 사회과학 계열 대학생, 대학원생
- 연구자: 정치학, 행정학, 정책학, 사회학 분야 연구자
- 일반 시민: 정치와 사회 이슈에 관심 있는 사용자

### 2-4. 사용자 문제

- 국회 회의록은 분량이 많아 필요한 발언과 맥락을 찾는 데 시간이 오래 걸린다.
- 키워드 검색만으로는 정책 의제의 흐름과 쟁점 구조를 파악하기 어렵다.
- 특정 의원, 정당, 정부 부처, 기관의 입장을 비교하려면 여러 회의록을 직접 확인해야 한다.
- 정책 실무자, 기자, 연구자는 인용 가능한 원문 근거가 필요하다.
- 일반 시민은 회의록 형식과 정책 용어 때문에 접근하기 어렵다.

### 2-5. MVP 핵심 기능 6개

- 정책 의제 분석
- 행위자별 발언 분석
- 쟁점 구조화
- 입장 비교
- 시계열 흐름 보기
- 원문 근거 확인

### 2-6. MVP 보강 레이어

MVP 6개 기능만으로는 "챗봇"에 머물 수 있다. GovTech 업무 도구가 되기 위해 다음 3개 레이어를 함께 고려한다.

```text
신뢰 레이어:
답변 신뢰도, 출처 커버리지, 확인 불가 안내

업무 산출물 레이어:
요약 복사, 출처 포함 복사, 브리핑 보기, 쟁점 표 보기

분석 템플릿 레이어:
정책 의제, 행위자, 입장 비교, 시계열 분석 템플릿
```

### 2-7. 데이터 범위와 중립성 원칙

정치·정책 서비스는 답변의 범위와 한계를 명확히 보여줘야 한다.

- 현재 분석 대상 위원회, 기간, 자료 유형을 표시한다.
- 외부 뉴스나 논평이 포함되지 않았는지 구분한다.
- 특정 정당, 정치인, 기관에 대한 가치판단을 단정하지 않는다.
- 회의록에 확인된 사실과 서비스의 분석·해석을 분리한다.
- 사용자가 원문 근거를 직접 확인할 수 있게 한다.

### 2-8. 기획 단계 완료 기준

- 서비스 목적이 명확하다.
- 타깃 사용자가 정의되어 있다.
- MVP 6개 기능이 정리되어 있다.
- 하지 않을 것이 정리되어 있다.
- 신뢰, 중립성, 데이터 범위 원칙이 포함되어 있다.

## 3. 서비스 설계

### 3-1. 설계의 의미

설계는 "어떻게 만들 것인가"를 정하는 단계다.

비전공자식으로 말하면, 설계는 개발 전에 만드는 지도다.

### 3-2. 설계 전체 구조

```text
화면 설계
데이터 설계
RAG/AI 설계
API 설계
신뢰 설계
배포·운영 설계
```

### 3-3. 화면 설계

사용자가 서비스를 어떻게 이용할지 정한다.

```text
첫 화면
→ 질문 입력
→ 위원회/기간/행위자 필터
→ 답변 확인
→ 출처 확인
→ 쟁점 표 또는 브리핑 보기
→ 복사 또는 저장
```

주요 화면:

- 질문 화면
- 답변 화면
- 출처 확인 패널
- 쟁점 표 보기
- 브리핑 형식 보기
- 원문 발언 보기
- 데이터 범위 안내 영역

### 3-4. 데이터 설계

회의록을 AI가 찾고 분석할 수 있도록 구조화한다.

핵심 데이터:

```text
위원회
회의
발언자
발언 원문
정책 의제
쟁점
출처
사용자 질문
답변 기록
피드백
```

기본 테이블:

```text
committees
meetings
speakers
chunks
embeddings
topics
query_logs
feedback
```

### 3-5. RAG/AI 설계

사용자의 질문이 들어온 뒤 답변이 만들어지는 흐름을 설계한다.

```text
사용자 질문 입력
→ 질문 유형 파악
→ 관련 회의록 검색
→ 발언자/날짜/위원회 확인
→ 정책 의제와 쟁점 구조화
→ 답변 생성
→ 출처 번호 연결
→ 근거 부족 여부 검사
→ 최종 답변 표시
```

### 3-6. API 설계

프론트엔드와 백엔드가 주고받을 약속을 정한다.

```text
POST /query
GET /committees
GET /meetings
GET /speakers
GET /citations/{id}
POST /feedback
GET /health
```

### 3-7. 신뢰 설계

GovTech 서비스에서는 신뢰 설계가 핵심이다.

- 답변 신뢰도 표시
- 출처 개수 표시
- 데이터 범위 표시
- 사실과 해석 구분
- 근거 부족 시 확인 불가
- 중립성 원칙 표시
- 원문 바로 확인

### 3-8. 설계 단계 완료 기준

- 화면 흐름이 정리되어 있다.
- 데이터 구조가 정리되어 있다.
- RAG 처리 흐름이 정리되어 있다.
- API 초안이 있다.
- 신뢰·중립성 원칙이 설계에 반영되어 있다.
- 배포 가능한 구조를 고려했다.

## 4. 개발

### 4-1. 개발의 의미

개발은 설계한 내용을 실제 서비스로 구현하는 단계다.

비전공자식으로 말하면, 설계도에 따라 실제 건물을 짓는 과정이다.

### 4-2. 개발 전체 구조

```text
개발 환경 준비
데이터 구축
데이터베이스 설계 및 저장
백엔드 개발
RAG/AI 개발
프론트엔드 개발
신뢰/검증 기능 개발
업무 산출물 기능 개발
테스트 및 배포 준비
```

### 4-3. 개발 환경 준비

프로젝트를 다음처럼 구성한다.

```text
project/
├── frontend/
├── backend/
├── data/
├── scripts/
├── docs/
├── tests/
├── .env.example
└── README.md
```

추천 기술 스택:

```text
프론트엔드: React + Vite
백엔드: FastAPI
데이터베이스: PostgreSQL + pgvector
AI 답변: OpenAI API
임베딩: OpenAI Embedding API 우선 검토
배포: Vercel + Render/Fly.io/Railway/Supabase 계열 검토
```

### 4-4. 데이터 구축

국회 회의록 원문을 검색 가능한 데이터로 바꾼다.

```text
원본 회의록 수집
→ 텍스트 추출
→ 텍스트 정리
→ 발언자 단위로 나누기
→ 메타데이터 붙이기
→ 품질 검사
→ 임베딩 생성
→ DB 저장
```

각 청크에 저장할 정보:

```text
chunk_id
원문 발언
발언자
소속 또는 역할
위원회
회의명
회의 날짜
정책 의제 후보
원문 파일명
원문 위치
```

### 4-5. 데이터베이스 개발

데이터베이스는 회의록과 검색 정보를 저장하는 공간이다.

MVP에서 필요한 기능:

- 위원회별 필터
- 날짜별 필터
- 발언자별 필터
- 의미 검색
- 키워드 검색
- 출처 원문 조회
- 질문 로그 저장

### 4-6. 백엔드 개발

백엔드는 사용자의 질문을 받아 RAG를 실행하고 답변과 출처를 반환한다.

백엔드가 하는 일:

```text
질문 받기
필터 받기
DB에서 관련 회의록 검색하기
AI 답변 생성 요청하기
출처 정리하기
신뢰도 계산하기
프론트엔드에 결과 보내기
질문 로그 저장하기
피드백 저장하기
```

### 4-7. RAG/AI 개발

RAG는 AI가 자기 기억만으로 답하지 않고 실제 회의록을 찾아보고 답하게 만드는 방식이다.

처리 흐름:

```text
질문 입력
→ 질문 유형 분류
→ 검색 쿼리 만들기
→ 관련 회의록 검색
→ 검색 결과 재정렬
→ 답변에 넣을 근거 선택
→ 분석 템플릿 적용
→ AI 답변 생성
→ 출처 번호 연결
→ 신뢰도 검사
→ 최종 답변 반환
```

질문 유형:

- 정책 의제 분석
- 행위자 분석
- 쟁점 구조화
- 입장 비교
- 시계열 흐름
- 원문 근거 확인
- 확인 불가 질문

### 4-8. 프론트엔드 개발

프론트엔드는 사용자가 직접 보는 화면이다.

주요 컴포넌트:

```text
QuestionInput
FilterBar
AnswerPanel
TrustBadge
CitationList
SourceDrawer
BriefingView
IssueTable
CopyButton
```

### 4-9. 신뢰/검증 기능 개발

정치·정책 도메인에서 안전하게 쓰이도록 신뢰 기능을 구현한다.

- 답변 신뢰도 표시
- 출처 커버리지 표시
- 데이터 범위 표시
- 근거 부족 안내
- 사실/해석 구분
- 원문 발언 확인
- 사용자 피드백

### 4-10. 업무 산출물 기능 개발

답변을 실제 업무에 가져다 쓸 수 있도록 만든다.

- 요약 복사
- 출처 포함 복사
- 브리핑 형식 보기
- 쟁점 표 보기
- 인용 발언 모아보기

### 4-11. 개발 단계 완료 기준

- 최소 1개 위원회 데이터가 검색된다.
- 사용자가 질문하면 답변이 생성된다.
- 답변에 출처가 붙는다.
- 원문 발언을 확인할 수 있다.
- MVP 6개 기능의 기본 질문이 처리된다.
- 근거 부족 질문에 확인 불가로 답한다.

## 5. 소스관리

### 5-1. 소스관리의 의미

소스관리는 코드와 문서의 변경 이력을 관리하는 과정이다.

비전공자식으로 말하면, 프로젝트의 작업 기록장이다.

### 5-2. 사용할 도구

```text
Git
GitHub
Issue
Pull Request
Release/Tag
```

### 5-3. 브랜치 전략

```text
main
- 배포 가능한 안정 버전

develop
- 다음 배포를 준비하는 개발 통합 버전

feature/*
- 기능별 작업 브랜치

fix/*
- 버그 수정 브랜치

docs/*
- 문서 작업 브랜치
```

MVP 단계에서는 `main`과 기능별 브랜치만 구분해도 충분하다.

### 5-4. 커밋 규칙

```text
feat: 새 기능 추가
fix: 버그 수정
docs: 문서 수정
refactor: 구조 개선
test: 테스트 추가/수정
chore: 설정, 빌드, 기타 작업
```

예시:

```text
feat: 질문 API 기본 응답 구조 추가
feat: 출처 원문 조회 기능 추가
fix: 발언자 필터가 적용되지 않던 문제 수정
docs: 서비스 기획서 초안 추가
```

### 5-5. 민감정보 관리

GitHub에 올리면 안 되는 것:

```text
OpenAI API Key
DB 비밀번호
DATABASE_URL 전체 값
배포 서비스 Secret
개인정보
```

관리 방식:

- 실제 비밀값은 `.env`에 넣는다.
- GitHub에는 `.env.example`만 올린다.
- `.gitignore`에 `.env`를 추가한다.
- 배포 서비스에는 환경변수로 등록한다.

### 5-6. 소스관리 완료 기준

- GitHub 저장소가 정리되어 있다.
- README가 있다.
- `.env.example`이 있다.
- 브랜치와 커밋 규칙이 정리되어 있다.
- Issue로 작업을 관리할 수 있다.
- 배포 가능한 버전을 Tag 또는 Release로 표시할 수 있다.

## 6. 배포

### 6-1. 배포의 의미

배포는 내 컴퓨터에서만 실행되던 서비스를 인터넷에 올려 다른 사람이 접속할 수 있게 만드는 과정이다.

### 6-2. 배포 목표

- 사용자가 링크로 서비스에 접속할 수 있다.
- 질문을 입력하면 실제 백엔드가 응답한다.
- DB에 저장된 회의록을 검색할 수 있다.
- 답변, 출처, 원문 확인이 정상 동작한다.
- API 키와 DB 비밀번호가 안전하게 관리된다.

### 6-3. 추천 배포 구조

```text
사용자
→ Vercel 프론트엔드
→ FastAPI 백엔드
→ PostgreSQL/pgvector DB
→ OpenAI API
```

후보 서비스:

```text
프론트엔드: Vercel, Netlify
백엔드: Render, Fly.io, Railway, Cloud Run
데이터베이스: Supabase Postgres, Neon, Railway Postgres, Render Postgres
```

### 6-4. prototype에서 바꿔야 할 점

```text
기존:
로컬 BGE-M3 모델
로컬 DB
로컬 백엔드

새 버전:
OpenAI Embedding API 또는 경량 임베딩
클라우드 Postgres
클라우드 FastAPI 백엔드
Vercel 프론트엔드
```

### 6-5. 배포 전 체크리스트

```text
[ ] 프론트엔드 빌드 성공
[ ] 백엔드 서버 실행 성공
[ ] DB 연결 성공
[ ] /health 정상 응답
[ ] /query 정상 응답
[ ] 출처 원문 조회 가능
[ ] 환경변수 등록 완료
[ ] CORS 설정 완료
[ ] README 업데이트
[ ] 데모 질문 준비
```

### 6-6. 배포 완료 기준

- 프론트엔드 배포 링크가 있다.
- 백엔드 API가 클라우드에서 동작한다.
- DB가 클라우드에서 연결된다.
- 사용자가 링크로 접속해 질문할 수 있다.
- 답변, 출처, 원문 확인이 가능하다.

### 13-1. 데이터 구축 단계

#### PDF 텍스트 추출

```text
위험: 국회 회의록 PDF는 스캔본이거나 복사 방지가 걸려 있는 경우가 있다.
     pdfplumber로 추출 시 일부 페이지가 빈 문자열로 나온다.
대응: 추출 직후 빈 텍스트 비율을 확인한다. 5% 이상이면 해당 PDF를 수동으로 확인한다.
```

```text
위험: 회의록마다 형식이 다르다.
     19대는 "◯", 20대는 "○", 21대는 "◎"처럼 발언자 마커가 조금씩 다른 경우가 있다.
대응: 실제 PDF를 먼저 열어보고 마커 패턴을 확인한 뒤 청킹 코드를 짠다.
```

```text
위험: 페이지 번호, 날짜 헤더, 법률 조문이 발언 내용과 섞여서 추출된다.
대응: 정규화 단계에서 제거할 패턴 목록을 별도로 만들어 관리한다.
```

#### 발언자 추출

```text
위험: "여러 위원", "야당 의원", "국민의힘 의원들"처럼 특정 인명이 아닌 표현이
     발언자로 추출되면 검색 시 0건이 반환된다.
대응: 추출된 발언자 목록에서 2-3자 한국 인명 패턴이 아닌 것을 걸러내는 검증 함수가 필요하다.
     "_is_valid_person_name_kw()" 같은 이름 검증 레이어를 처음부터 설계에 포함한다.
```

```text
위험: 당명이 이름으로 오인된다.
     "국민의힘 의원" → "의힘"이 2자 인명으로 추출되는 경우가 발생한다.
대응: 당명 접미어(의힘, 민당, 주당 등) 블랙리스트를 만들거나,
     이름 직전 문자가 한글이면 매칭을 실패시키는 lookbehind 패턴을 쓴다.
```

#### 메타데이터 품질

```text
위험: 회의 날짜 추출 실패.
     날짜가 PDF 헤더에 있는 경우와 본문 중간에 있는 경우가 섞여 있다.
대응: 추출 직후 meeting_date null 비율을 출력한다.
     null이 5% 이상이면 해당 파일들을 확인한다.
```

### 13-2. 데이터베이스 단계

#### pgvector 설치

```text
위험: 클라우드 Postgres 서비스에 pgvector 확장이 기본 설치되지 않은 경우가 있다.
대응: DB 서비스를 선택하기 전에 pgvector 지원 여부를 확인한다.
     Supabase, Neon, Railway, Render Postgres는 모두 지원한다.
     일반 RDS는 수동 설치가 필요하다.
```

```text
위험: 임베딩 모델을 바꾸면 벡터 차원이 달라진다.
     OpenAI text-embedding-3-small: 1536차원
     OpenAI text-embedding-3-large: 3072차원
     이미 만들어진 테이블의 차원은 바꿀 수 없다.
```

### 14-1. 검색 관련

#### BM25와 벡터 검색은 반드시 함께 써야 한다

벡터 검색(Dense)만 쓰면 발언자 이름, 날짜, 고유명사처럼 정확한 단어 매칭이 필요한 쿼리에서 결과가 나쁘다.  
"조태열 장관이 한 발언" → 의미가 비슷한 다른 발언자 결과가 상위로 올라온다.

BM25와 벡터 검색을 RRF(Reciprocal Rank Fusion)로 결합하면 두 검색의 장점이 더해진다.  
이것을 나중에 추가하면 임베딩부터 검색까지 많은 코드를 고쳐야 한다.  
처음부터 하이브리드 검색 구조로 설계한다.

#### 발언자 필터와 집계 쿼리 감지는 서로 충돌한다

"어떤 의원들이 X를 주장했나?" → 발언자 필터를 적용하면 speaker='어떤' → 0건.

발언자 필터를 적용하기 전에 집계 쿼리 여부를 먼저 확인해야 한다.  
집계 쿼리(여러 발언자 요약)와 개인 발언자 쿼리는 처리 방식이 다르다.  
집계 쿼리에서는 발언자 필터를 끄고 다양한 발언자가 나오도록 balance_speakers를 켠다.

#### FTS OR 쿼리

PostgreSQL의 plainto_tsquery는 AND 조건으로 동작한다.  
"조태열 장관 한미동맹" → 세 단어가 모두 포함된 문서만 반환 → 거의 0건.

발언자 이름은 embed_text 앞에 붙이는 prefix에만 있고 clean_text에는 없는 경우가 많다.  
FTS가 clean_text만 보면 이름 포함 쿼리에서 항상 0건이 나온다.

해결책: 토큰을 추출해서 OR로 연결한다.
```python
tokens = re.findall(r"[가-힣a-zA-Z0-9]{2,}", query)
tsquery = " | ".join(tokens)
```

### 14-2. LLM 생성 관련

#### 프롬프트에 언어 규칙을 명시해야 한다

프롬프트에 언어 규칙이 없으면 LLM이 한국어 답변 중간에 영어 단어를 섞는다.  
"reaffirm", "signify", "underscore" 같은 단어가 갑자기 나온다.

해결책: 시스템 프롬프트에 "고유명사·약어(UN, NATO) 외 영어 단어를 본문에 혼용하지 않는다"를 명시한다.

#### 히스토리 전달 시 인용 블록을 제거해야 한다

멀티턴 대화에서 이전 답변을 그대로 히스토리에 넣으면 JSON 인용 블록까지 함께 전달된다.  
인용 블록 1개가 수천 토큰이라 히스토리 3턴이면 컨텍스트 상당 부분을 차지한다.

해결책: 히스토리를 만들 때 `<!--RAG_REFERENCES-->` 이하를 잘라내고 전달한다.

#### REFUSED 레벨을 처음부터 설계에 넣어야 한다

할루시네이션 트랩(존재하지 않는 인물, 허구 문서 등)에 대한 올바른 거절은  
NONE(근거 부족)과 구별해서 처리해야 한다.

나중에 추가하면 eval 채점 로직, grounding_check, 프론트엔드 UI 표시를 전부 고쳐야 한다.  
처음부터 FULL / PARTIAL / REFUSED / NONE 4단계로 설계한다.

#### 존재 여부 감지 패턴은 좁게 만든다

"논의된 적이 있나요?", "발언한 내용이 있나요?" 같은 존재 여부 질문은  
더 강한 모델이나 2단계 검증이 필요하다.  
하지만 패턴이 너무 넓으면 일반 질문까지 걸려서 오히려 성능이 떨어진다.

실제 경험: "있나요" 단독 패턴을 추가했더니 "차이가 있나요?", "어떤 입장이 있나요?" 같은  
일반 질문이 전부 존재 확인으로 분류되어 정답률이 90% → 54%로 떨어졌다.

해결책: 의미 단위 콤보 패턴만 사용한다.
```python
r"발언한\s*내용이\s*있나요"
r"논의된\s*적이\s*있나요"
r"한\s*내용이\s*있나요"
```

### 14-3. 데이터 파이프라인 관련

#### 증분 임베딩을 처음부터 만들어야 한다

새 데이터를 추가할 때마다 전체를 다시 임베딩하면 비용과 시간이 크다.  
처음부터 이미 임베딩된 chunk_id를 확인하고 없는 것만 처리하도록 만든다.

```python
existing = {row[0] for row in db.execute("SELECT chunk_id FROM embeddings")}
to_embed = [c for c in chunks if c["chunk_id"] not in existing]
```

#### 품질 검사 없이 적재하면 나중에 디버깅이 어렵다

speaker=null, meeting_date=null인 청크가 검색 결과에 나타나면  
LLM이 발언자와 날짜 없이 답변한다.  
사용자 입장에서는 근거를 확인할 수 없다.

해결책: 적재 전에 품질 리포트를 출력하는 스크립트를 만든다.  
speaker 채움률, meeting_date 채움률, 빈 content 비율을 확인하고  
임계치 이하면 적재를 중단한다.

#### 위원회 필드를 처음부터 청크에 저장해야 한다

나중에 위원회 필터를 추가하려면 모든 청크에 committee 필드가 있어야 한다.  
처음에 넣지 않으면 전체를 다시 재적재해야 한다.

### 14-4. 배포 관련

#### 임베딩 모델 선택이 배포 가능 여부를 결정한다

로컬 임베딩 모델(BGE-M3 등)은 성능이 좋지만 RAM을 2GB 이상 사용한다.  
무료 클라우드 서버(Render 무료: 512MB, Railway 무료: 512MB)에서는 실행이 불가능하다.

포트폴리오 배포가 목표라면 OpenAI Embedding API를 써야 한다.  
API 호출 방식이라 서버 RAM이 거의 필요 없다.

#### Render 무료 티어의 슬립 문제

Render 무료 티어는 15분 동안 요청이 없으면 서버가 잠든다.  
포트폴리오 시연 중 면접관이 처음 접속하면 1분을 기다려야 하는 상황이 생긴다.

해결책: Render Basic($7/월) 이상을 쓰거나, UptimeRobot 같은 무료 핑 서비스로  
15분마다 /health를 호출해 잠들지 않게 한다.

### 14-5. 평가(eval) 관련

#### eval을 처음부터 만들어야 한다

개발 후반에 eval을 추가하면 이미 코드가 많이 바뀌어서 기준선을 잡기 어렵다.  
처음부터 10-20개 질문으로 작은 eval을 만들어두면  
코드를 바꿀 때마다 성능이 오르는지 내리는지 확인할 수 있다.

#### grounding_ok 기준을 명확히 정해야 한다

FULL과 PARTIAL을 둘 다 성공으로 볼 것인가, FULL만 성공으로 볼 것인가에 따라  
숫자가 크게 달라진다.  
이 기준을 처음에 정하지 않으면 나중에 수치 비교가 불가능해진다.

#### unanswerable 문항은 별도로 채점해야 한다

할루시네이션 트랩 질문은 거절이 정답이다.  
grounding_ok와 같은 기준으로 채점하면 NONE이나 REFUSED가 실패로 잡힌다.  
처음부터 unanswerable 문항은 "올바른 거절 여부"로 별도 채점한다.

### 14-6. 전체적인 교훈

```text
1. 배포 가능성을 먼저 고려하고 기술을 선택한다.
   좋은 로컬 모델보다 배포 가능한 API 모델이 포트폴리오에서 낫다.

2. 임베딩 모델은 개발 시작 전에 확정한다.
   바꾸면 DB 스키마, ETL, 검색 코드를 모두 다시 만들어야 한다.

3. 데이터 품질 검사는 파이프라인의 일부다.
   품질 확인 없이 적재하면 검색 결과가 이상해도 원인을 찾기 어렵다.

4. 발언자 추출은 단순 regex로 끝나지 않는다.
   한국어 이름과 직함, 당명, 지시 관형사가 섞여서 예상치 못한 케이스가 계속 나온다.
   처음부터 이름 검증 함수를 만들고, 발견되는 예외를 블랙리스트에 추가해간다.

5. 하이브리드 검색은 처음부터 설계에 포함한다.
   나중에 추가하면 구조 변경이 크다.

6. eval 숫자가 내려가는 건 코드를 바꿨기 때문이다.
   회귀가 발생하면 가장 최근에 바꾼 것부터 되돌리며 원인을 찾는다.
   여러 개를 동시에 바꾸면 원인을 찾기 어려워진다.

7. 환경변수 관리는 첫날부터 한다.
   나중에 실수로 .env를 커밋하면 API 키를 즉시 교체해야 한다.
```
